# ワークフロー定義

OS の5つのコアワークフロー。ユーザが名前で指示したら即実行する。各ワークフローの担当（OS / Hermes）と出力先を厳守する。前提ルールは [[OS-運用マニュアル]]。

---

## 1. Daily Morning Briefing

- **トリガー**：「Morning Briefing」「今日の状況」/ `Initialize OS`
- **担当**：OS（構造パート）＋ Hermes（私的内容の要約パート）
- **入力**：今日の `01_Daily/`、`02_Inbox/`、`05_Tasks/inbox` `active` `blocked`、`03_Projects/` の active、各ノートの `deadline`
- **手順**
  1. [[ダッシュボード]] のクエリ結果を取得（最新の構造データ）。
  2. 期限切れ・当日期限・今後3日の `deadline` を抽出。
  3. `blocked` タスク（ユーザ判断待ち）を最優先で提示。
  4. 日次ノートの自由記述の要約が必要な場合のみ Hermes にローカルで実行させ、その出力を読み込む。OS は日次ノート本文を直接外部要約しない。
  5. 自然言語で「今日の優先事項 / 進行中プロジェクト / 期限 / オープンループ / 判断待ち」を統合。
- **出力**：`07_Generated/briefings/YYYY-MM-DD-briefing.md`
- **ガード**：収益パルスはデータソース未接続のため、接続まで「N/A」と明記し推測しない。

---

## 2. Capture Processor

- **トリガー**：「Capture処理」「Inbox整理」
- **担当**：Hermes（一次トリアージ）＋ OS（レビューと実行）
- **入力**：`02_Inbox/` の全ノート
- **手順**
  1. Hermes が各アイテムを分類し、推奨 `type` / `tags` / 移動先フォルダを提案（ローカル処理）。
  2. OS が提案をレビューし、frontmatter 未編集（例：`priority: high | normal | low`）を検出して確定値を提案。
  3. 移動先候補と理由を一覧でユーザに提示。
  4. ユーザ承認後に移動を実行。承認前に勝手に移動しない。
- **出力**：分類提案リスト（チャット）。確定後、各ノートを該当フォルダへ移動。
- **ガード**：8B の分類は誤りうるため自動適用しない。判断に迷うアイテムは `02_Inbox/` に残す。

---

## 3. Weekly Review Generator

- **トリガー**：「Weekly Review」「週次レビュー」
- **担当**：OS
- **入力**：直近7日の `01_Daily/`、`05_Tasks/done/`、`03_Projects/` の active、前回レビュー
- **手順**
  1. 完了タスク・進んだプロジェクト・停滞項目を集計。
  2. オープンループの棚卸し（持ち越し・破棄・再優先）。
  3. 次週の優先事項（3〜5件）を第一原理で導出。
  4. CLAUDE.md / [[OS-運用マニュアル]] の優先事項に対する更新案を提示。
- **出力**：`07_Generated/reviews/YYYY-Www-review.md`
- **ガード**：CLAUDE.md 本体の更新はユーザ承認後に実行（提案止まり）。

---

## 4. Queue Processor

- **トリガー**：「Queue処理」「キューを回して」
- **担当**：`delegate` フィールドで分岐（`hermes3`→Hermes、`claude-*`→OS、`auto`→OS が判断）
- **入力**：`05_Tasks/inbox/` のタスクノート
- **手順**
  1. `inbox/` のタスクを `priority` 順に読む。
  2. `delegate` を見て担当を決定。タスクを `active/` へ移動。
  3. 実行。成果物は原則 `04_Resources/` 配下に保存。
  4. 完了したらタスクノートに結果サマリを追記し `done/` へ移動。
  5. 判断が要るものは `blocked/` へ移動し、`## 質問` セクションに論点を書く。
- **出力**：成果物（`04_Resources/`）＋ タスクノートの状態遷移
- **ガード**：法的判断・金銭の決定・個人情報を含む処理は必ず `blocked/`。Hermes 出力（特に事実主張）は OS が抜き取り検査する（過去にローカルLLMが人物を取り違えた事例あり）。

---

## 5. Project Health Monitor

- **トリガー**：「Project Health」「プロジェクト健康診断」
- **担当**：OS
- **入力**：`03_Projects/` の `status: active` の全ノート
- **手順**
  1. 各プロジェクトの最終更新日・`status`・「次のアクション」セクションを確認。
  2. 警告判定：14日以上更新なし＝停滞、「次のアクション」が空＝詰まり、`deadline` 超過＝遅延、`owner` 空＝責任者不在。
  3. 重大度順に警告を一覧化し、各項目に推奨アクションを添える。
- **出力**：`07_Generated/health/YYYY-MM-DD-health.md`
- **ガード**：プロジェクトノート本体は書き換えず、診断結果のみ出力する。
