01 — Vision
AIはまだ、あなたのことを知らない

今日のAIとの対話は、毎回が「初対面」だ。どれだけ優秀なモデルでも、あなたの価値観・習慣・記憶・人生の文脈を持たずに答えを返している。Ghost-Printerはその根本を変える。

現在 — 一元的・瞬間的
›テキスト入力だけがAIへのインプット
›文脈はその会話の中だけに存在
›毎回「自分」を説明しなおす
›データはクラウドに預けるしかない
›誰が何を見るか管理できない
→
Ghost-Printer — 多元的・継続的
›音声・行動・感情・場所が複合的に積み重なる
›長年の文脈がAIに渡せる
›AIはあなたを「知っている」状態で答える
›完全オフライン動作・ローカル保存
›何を誰に見せるか自分が決める

Ghost-Printerは「魂のプリンター」だ。日常の無数の瞬間から、その人固有のパターン・価値観・記憶を静かに抽出し、AIが読めるフォーマットに印刷し続ける。生成するのでなく、記録する。それがこのデバイスの本質。

02 — SOUL Format
SOULとは何か

SOULはAIが読み込むための人格ファイルだ。人間が読む必要はない。確率分布・埋め込みベクトル・隠れマルコフモデルで表現される「AI-native」なフォーマット。情報密度は自然言語の説明より桁違いに高い。

SOULの4層構造
Core Identity
変化が遅い・安定した自己
personality_prior多次元Gauss分布 (128dim)
values[]Dirichlet分布
communication_styleembedding vec
Episodic Memory
3層: 直近・圧縮・蒸留
recent (30日)vec[512] + weight
compressed (180日)GMM centroids
distilled (180日+)→ prior に溶ける
Semantic Map
興味・人間関係・スキル
topic_dirichletfloat32[512]
people_graphsparse adj matrix
Temporal Patterns
行動の時系列リズム
hmm_states隠れマルコフモデル
circadian[24]時間帯別エネルギー
ベイズ的忘却 — エピソードが性格に溶ける
記憶を「消す」のではなく、重要度の低いエピソードは時間とともに確率分布の更新として吸収される。繰り返し観測されたパターンは σ（不確かさ）が縮小し、「性格特性」として定着する。
# エピソードが積み重なるにつれて # 性格の確信度が上がる Day 1 introversion: μ=0.72, σ=0.28 Day 30 introversion: μ=0.72, σ=0.14 Day 180 introversion: μ=0.72, σ=0.06 # エピソード削除。性格として定着。 # SOULファイルはむしろ軽くなる。
ローカルモデルスタック（完全オフライン）
Whisper tiny
39 MB
役割音声 → テキスト
Bonsai 1.7B
240 MB
役割意味抽出 / 重要度判定
MiniLM-L6
22 MB

03 — Hardware
Core + Shell アーキテクチャ

Coreは販売品。Shellはコミュニティが育てるOSS。機能を絞ったCoreが信頼の根拠になる。

Core — 最小機能セット（販売品）
SoCCortex-A53 class
RAM512MB〜1GB
StorageeMMC 8GB
Battery IC込み
BLE 5.0込み
MicrophoneI2S MEMS 1基
ConnectorUSB-C or Pogo ×4
Displayなし
Camera / GPSなし → Shell へ
機能を絞ることが攻撃面の縮小になる。
信頼の根拠はシンプルさにある。
Shell — OSS / 3Dプリンタ対応
⌚ Wearable
常時装着。Pogo Pinドック充電。心拍・IMU・GPSをShellに搭載。
🖥 Desk
据え置き。USB-C接続。会議・作業をパッシブに記録。環境センサーも可。
💎 Pendant
アクセサリー型。Pogo Pin専用小型ドック。最小存在感を目指す。
🚗 Car
車内常設。USB-C電源。移動・通勤パターンを記録。
Soul Dock（充電 = 同期 = 認証）
毎晩ドックに置くだけで全Coreが高帯域同期。新しいCoreを挿すだけで信頼確立が完了する。ドック自体がHome Coreとして機能する。
Core-to-Core 信頼確立（有線ファースト）
STEP 1
物理接続
USB-C / Pogo で接続。物理的所持が証明になる。
STEP 2
ECDH鍵交換
有線上でMITM不可能な鍵交換を実行。
STEP 3
初回フルマージ
USB 2.0 (480Mbps) で SOULを全量同期。
以降
BLE 差分同期
信頼済み。近づくと自動で差分のみ同期。
複数CoreはCRDT（競合なし自動マージ）で同一SOULに収束。オフライン中の更新も矛盾なく統合される。
04 — Soul Protocol
オープンプロトコルとしての拡張性

Core同士・サードパーティアプリ・外部AIが接続できるオープンスタック。ActivityPubのPersonal Identity版。

L5
Plugin Layer
任意のデバイス・アプリがSOULにデータを寄与できる拡張I/F。OSS仕様として公開。3種類（Source / Lens / Bridge）。
soul-plugin spec
L4
Merge Layer
複数観測をBayesian fusionで統合。矛盾は「両立する複合状態」として保存。
CRDT + Bayes
L3
Permission Layer
Capability Tokenによるスコープ制御。カテゴリ・期間・粒度の3軸で開示範囲を設定。
capability tokens
L2
Sync Layer
SOUL fragmentの差分同期プロトコル。有線480Mbps（初回）・BLE 1Mbps（差分）。
delta-sync
L1
Discovery Layer
BLEアドバタイジングで近傍Coreを発見。soul_idハッシュで同一人物のCoreを識別。
BLE adv
05 — Selective Disclosure
選択的自己開示

誰に・何を・どこまで見せるかを自分が決める。開示はカテゴリ・期間・粒度の3軸で制御される。

開示スコープ（Claude Personalの例）
性格・価値観
Fullコアアイデンティティ
エピソード記憶
直近90日
行動パターン
Full
バイタル・健康
サマリーのみ
位置・行動履歴
非開示
人間関係グラフ
匿名化
外部AIへのインターフェース
# Permission API エンドポイント GET /soul/summary # 認証不要 GET /soul/identity # Token必須 GET /soul/memory?scope= # スコープ制限 GET /soul/context/now # 直近の状態

# Ghost-Printer が自動生成 あなたは {name} のアシスタントです。 以下はその人のSOULデータです： personality: {soul.identity} recent_context: {soul.context} patterns: {soul.patterns} 開示スコープ外の情報は含まれていません。
06 — Next Steps
次に進むべき3つのトラック

Ghost-Printerを実装に向けて進めるにあたり、並行して進められる実験・設計のトラックを定義する。

A
Track A — Experimentation
SOUL抽出方法の実験
どのような情報をどのように扱うか。データパイプラインとフォーマットを実際に動かして検証する。ハードウェアは不要、PC上でプロトタイプ可能。
A1
手動入力 → SOUL生成パイプライン
日記・メモをテキストで入力 → Bonsai 1.7B が感情・重要度・価値観を抽出 → .soul ファイルに書き込む。最もシンプルなループを最初に動かす。
Week 1-2
A2
抽出プロンプトの精度検証
「カフェで一人でコーヒーを飲んだ」→「内向性: 0.72」を引き出せるか。重要度スコアの妥当性を人手で評価。プロンプトをイテレーションする。
Week 2-3
A3
多元インプットの統合実験
音声（Whisper）+ 位置（手動）+ 時刻を束ねたコンテキストをBonsaiに渡す。単一インプットと比較してSOULの質がどれだけ向上するかを測定。
Week 3-4
A4
ベイズ的忘却アルゴリズムの実装
エピソードの重みを時間減衰させ、personality priorに吸収するロジックを実装。「記憶を消す」ではなく「溶け込む」が正しく動くか検証。
Week 4-5
A5
Claude連携テスト
生成したSOULをSystem Promptに変換 → Claudeに読み込ませる。「このSOULを知っているClaudeは何が違うか」を体験として検証。
Week 5-6
B
Track B — Hardware Design
物理的な基盤・製品の設計
CoreボードとShellの設計。OSSとして公開できる形にする。まずは既製品（Raspberry Pi）で動作を確認し、徐々に専用基板へ移行する。
B1
Pi Zero 2W でのスタック動作確認
Whisper tiny + Bonsai 1.7B + MiniLM が Pi Zero 2W（RAM 512MB）で実用的に動くか検証。メモリ・スワップ・速度を計測。必要ならPi 5（2GB）に変更。
Early
B2
バッテリーHATの選定と電力設計
Tier1（常時低消費）・Tier2（トリガー起動）・Tier3（バッチ処理）の消費電力を実測。バッテリー容量と実稼働時間を算出する。
Mid
B3
Soul Dockのプロトタイプ（3Dプリンタ）
Pogo Pin 4極（USB 2.0互換）を使ったドックを設計・印刷。Coreを置くだけで充電・同期が完了する体験を作る。STLファイルをGitHubで公開。
Mid
B4
Coreボードの回路設計（KiCad）
Pi Compute Module相当の専用基板設計。SoC・電源IC・BLEモジュール・MEMS Micの最小構成を回路図に落とす。Gerberファイルを公開。
Later
B5
Shell設計仕様の策定とコミュニティ公開
CoreへのShell取り付け規格（機械的・電気的インターフェース）を仕様化。Printablesに第1世代Shellデザインを投稿してコミュニティを立ち上げる。
Later
C
Track C — Disclosure Design
選択的自己開示の設計
何をどのように分け、外部AIに共有するかを設計する。これはGhost-Printerの「信頼の哲学」そのものであり、最も慎重に考えるべきトラック。
C1
開示カテゴリの定義と分類実験
SOUL内のデータをどのカテゴリに分けるかを決める。「性格」「記憶」「行動」「健康」「関係」の5軸が基本。カテゴリ境界が曖昧なケース（例：習慣は性格か行動か）を洗い出す。
Early
C2
Capability Token仕様の設計
外部AIがSOUL APIにアクセスする際のトークン設計。スコープ（カテゴリ）・期間・粒度（全文 / サマリー / 匿名）を組み合わせて表現できるフォーマットを定義する。
Mid
C3
「どこまで見せると何が変わるか」の体験検証
開示スコープを変えながらClaudeと会話し、どこまで共有すると何が変わるかを体験的に記録する。「健康データを加えると提案の質が変わるか」など具体的なケースで検証。
Mid
C4
管理UIの設計（スマホアプリ）
Coreに画面がないため、開示設定はスマホアプリで行う。「AIごとにスコープを設定」「一時的な共有（今日だけ）」「緊急停止（全開示を即時取り消す）」の3機能を最初に設計する。
Mid-Late
C5
他者間共有プロトコルの設計
パートナー・家族・医師など、別の人間にSOULの一部を共有するフロー設計。「同意の取得」「開示範囲の合意」「取り消し」をどう設計するかは、単なるAPI設計を超えた哲学的判断を含む。
Later
07 — Open Questions
未解決の設計上の問い
未決
マイクの常時起動は法的・倫理的に許容されるか？
「VADで検知のみ・録音はしない」か「ユーザーが明示的に開始」かを決める必要がある。地域によって法的要件が異なる。
未決
BLE帯域でSOUL差分同期は現実的か？
BLEは実効1Mbps程度。SOULが大きくなると転送がボトルネックに。差分設計の精度と、有線フルマージの頻度で対応する想定。
要議論
「デジタル人格」としての利用で生じる倫理的問題
死後のSOULはどうなるか。AIがその人を「演じる」ことの境界はどこか。プロダクトとして早期に立場を明確にすべきトピック。
要設計
CoreのOSとソフトウェアのライセンス
BonsaiはApache 2.0。Whisper / MiniLMも開放的ライセンス。Ghost-Printer自体のOSS戦略（Core OSをどこまで公開するか）を定義する必要がある。
